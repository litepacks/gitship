GitHub-Driven Deployments

Alpha

GitShip

A lightweight, self-hosted deployment toolkit designed to sync local project settings, register GitHub webhooks on-the-fly, and deploy repositories on push events.

Continuous Integration/Deployment without complex dashboards or cloud platforms. Targets Node.js 22+.

Why GitShip?

Platforms like n8n or Coolify offer great self-hosted control, but they often come with heavy visual dashboard footprints and setup complexity. GitShip is a CLI-first alternative built specifically for developers who want a GitOps flow with zero bloat: configure a single gitship.yml file, run `sync`, and let GitShip manage hook lifecycles and build logs automatically.

Ideal when...

  • You want simple deployments triggered by a Git push
  • You prefer configuration-as-code (committed directly to the repo)
  • You deploy on a VPS, local server, or remote instances via SSH
  • You need clean real-time logs directly in your shell

Architectural Highlights

  • Timing-safe webhook signature validations (HMAC SHA-256)
  • FIFO deployment queue with process monitoring & cancellation
  • Automatic Node version management wrapping via NVM
  • Zero external databases required — stores state in a local JSON file

Features

🔒 Secure Webhooks

Timing-safe signature verification protects your server endpoint against unauthorized payload executions.

🚂 FIFO Queue

Queued deployment runs process in order, preventing concurrency resource locks or duplicate operations.

🔗 SSH Daemons

Target local environments or execute pipeline operations on remote hosts securely using SSH commands.

⏳ Rollbacks

Easily roll back deployments to any past successful commit hash with a single command.

📦 NVM Integration

Declare project-level Node.js versions in config and let GitShip handle installations on demand.

🩺 Health Checks

Define test endpoints to verify deployment stability and health automatically upon restart.

Installation

Install the global GitShip CLI binary on your terminal host. Make sure Node.js v22+ is active.

npm install -g gitship-cli

Quick Start

1. Authenticate with GitHub

Authenticates your CLI, using standard browser redirects or manually inputting a personal access token.

gitship auth github

2. Initialize a Project

Detects the current project type (PM2, Docker, Node) and prompts for targets to generate your gitship.yml template.

gitship init

3. Sync Configuration and Setup Webhook

Validates configuration schemas, creates the local project database records, generates timing secrets, and updates webhook settings on GitHub automatically.

gitship sync

Configuration Reference — gitship.yml

Here is a complete schema configuration reference for the gitship.yml file:

# Unique identifier name for the project
project: landing-page

# Repository reference coordinates
repository:
  owner: my-organization
  repo: primary-site
  branch: main

# Deployment destination target parameters
target:
  type: ssh                  # 'ssh' or 'local'
  host: 192.168.1.10:2222    # (Required for ssh) format IP:Port or host:port
  path: /var/www/landing     # Destination target deployment directory path

# Deployment commands & steps configuration
deploy:
  node_version: "22.2.0"     # Optional: wraps commands using specific node version (via nvm)
  install: npm ci            # Dependency install command
  build: npm run build       # Build project command
  restart: pm2 restart all   # Post-deployment restart command
  
  # Health checks configuration (Optional)
  healthcheck:
    path: /health            # Query endpoint (default: '/')
    port: 3000               # Port to check
    retries: 5               # Max check retries (default: 5)
    interval_ms: 1000        # Interval between checks (default: 1000)
    timeout_ms: 2000         # Single check request timeout (default: 2000)

# Build metrics logging options (Optional)
logging:
  enabled: true              # Persist run stdout/stderr logs in database (default: true)

Node Version Management (NVM)

GitShip supports project-level Node.js isolation via NVM (Node Version Manager). The tool automatically detects local NVM configurations and wraps build scripts.

1. Developer Workspace Setup

To configure your local workspace node environments, a .nvmrc is provided in the repository root. Run the following: 

# Install specified version
nvm install

# Use project Node version
nvm use

2. Execution Wrapping in gitship.yml

When the deploy.node_version field is configured in your project's YAML configuration, GitShip ensures isolated execution pipelines:

  • Automatic Wrapping: Commands (`install`, `build`, `restart`) are executed under the declared Node version using environment wrapper scripts.
  • Auto-Installation: If the target environment (local or SSH VPS host) does not have the specified version installed, GitShip runs nvm install <version> on-the-fly.
  • Discovery Paths: Auto-detects NVM install scripts inside ~/.nvm, /usr/local/nvm, and Homebrew paths.

CLI Command References

Command Description
gitship auth github Authenticate with GitHub via browser OAuth redirect or personal token.
gitship init Initialize a new project and create a local gitship.yml.
gitship sync Sync local configurations with database records and configure GitHub webhooks.
gitship check Check if local configuration, database record, and remote GitHub webhooks are valid.
gitship runs List recent deployment runs and statuses.
gitship logs <run-id> [-f] Print build output logs for a run. Stream live outputs with --follow.
gitship rollback <run-id> Queue a rollback deployment reverting target to the specified commit hash.
gitship stats Show project metrics, average build times, and success rates.
gitship queue Display currently active or queued deployment runs.
gitship cancel <run-id> Cancel a pending run in the queue or terminate active processes.
gitship project inspect <name> Inspect configuration details and status for a registered project.
gitship repos List all repositories in the connected GitHub account.

Database Schema

GitShip maintains states locally in `~/.gitship/gitship.json`. The JSON file is divided into five core collections:

📂 projects

Contains active project properties, target specifications (local/SSH paths), repository mappings, and webhook secrets used to sign payload deliveries.

🔗 webhooks

Saves webhook metadata, active state flags, delivery secret values, and the remote ID returned by the GitHub Hooks API.

🏁 deployments

Tracks overall statuses (QUEUED, RUNNING, SUCCESS, FAILED, CANCELLED), commit SHA hashes, trigger author tags, and total time elapsed in milliseconds.

📊 deployment_steps

Holds timing performance data, statuses, and exceptions for specific deployment pipeline steps (clone, install, build, restart, healthcheck).

📝 deployment_logs

Full terminal execution outputs containing combined stdout and stderr recordings generated during build executions.

VPS Setup & Production Guide

When deploying GitShip to a production Virtual Private Server (VPS), coordinate your environment architecture according to these guidelines.

1. Local vs VPS Synchronization

Since GitShip stores project states locally under ~/.gitship/gitship.json, the Server Agent on the VPS needs access to the configuration records.

⚠️ Coordination Requirement: Webhooks point to the VPS Server Agent. If you run gitship sync locally on your developer machine, the local database gets populated, but the VPS database won't contain the record, causing incoming webhooks to fail with a project mismatch error.

To synchronize, choose one of the following methods:

  • Method A (Recommended): Run gitship init locally to generate gitship.yml, push it to your repository, SSH into the VPS, clone the project, and run gitship sync directly on the VPS.
  • Method B: Run gitship sync locally pointing the agent URL to your VPS, then manually copy the local ~/.gitship/gitship.json file to the VPS configuration directory.

2. Running Server Agent continuously via PM2

Keep the webhook server agent running continuously in the background on your VPS using PM2:

# Start the Server Agent daemon
pm2 start "gitship agent" --name "gitship-agent"

# Or on a custom port
pm2 start "gitship agent --port 8080" --name "gitship-agent"

# Setup PM2 daemon to restart on system boot
pm2 startup
pm2 save

3. SSL Certificates & GitHub OAuth

To use secure webhooks (HTTPS callbacks), route the Server Agent port behind a reverse proxy (e.g. Nginx) configured with Let's Encrypt certificates.

When authenticating the CLI directly on the VPS via SSH:

  • Select the Personal Access Token (Manual) option during auth, or
  • Connect over SSH using port forwarding: ssh -L 4567:localhost:4567 user@your-vps-ip to capture the browser redirect locally.

Development & Monorepo Workspace

GitShip is structured as an npm workspaces monorepo:

packages/shared

Shared schema validations (Zod), project types, and yaml config parsers.

packages/core

Database drivers, command execution engines, and FIFO deployment queue controllers.

packages/server-agent

Stand-alone Express webhook receiver verifying hub payload signatures.

packages/cli

Commander.js CLI providing commands for auth, init, runs, stats, and logs.

Development Commands

# Run unit tests
npm test

# Build all workspace packages
npm run build

# Run security audits
npm run audit

# Bump workspace package versions
npm run change patch
